# Saelis Her — Phase 1 Report (Foundation)

Date: 2026-07-17 · Status: complete · Not committed, not pushed, not deployed, no migration applied.

Phase 1 delivers the foundational architecture: CLAUDE.md, the pathway registry, migration 00007
(all tables, RLS, seeds), the type boundary, Zod schemas, the server-only service layer, and
tests. No dashboard, workout generator, nutrition planner, or companion integration was built —
those are later phases by design.

Shared (neutral naming): `wellness_pathways`, `wellness_enrollments`, `women_wellness_profiles`,
`wellness_goals`, `wellness_daily_check_ins`, `wellness_programs`, `wellness_program_weeks`,

Global libraries (read-only to users): `exercise_library`, `workout_templates`,
`workout_template_exercises`, `meal_templates`.

Notable structural decisions:

- `user_id` references `public.profiles(id)` (repo convention) rather than `auth.users(id)`
  directly — profiles cascades from `auth.users`, so account deletion cascades identically and
  the schema stays consistent with migrations 00001–00006.
- One ACTIVE enrollment per (user, pathway) enforced by a partial unique index; multiple
  simultaneously active pathways fully supported.
- Check constraints mirror every enumerated value and range in the spec (1–5 scales, 0–5
  soreness, 0–10 pain, calorie floor ≥ 1200, exertion 1–10, etc.).
- `medical_clearance_status` is commented as self-reported; the app never auto-sets `cleared`.
- Weight/calorie columns are nullable everywhere — tracking is optional by product rule.

throw calm content-free errors, and never log wellness or symptom values.

## Known limitations

- SQL is carefully constraint-checked but has not executed against a real Postgres yet.
- The type boundary is hand-written; regenerate after `db push` (drift is a bug).
- Icon identifiers in the registry have no UI mapping yet (no Her routes/components exist).
- Programs/daily plans/meal plans have write paths but no rules engine to generate them.
- The safety gate (red flags → `safety_hold`) is schema- and registry-level only; the
  deterministic rules engine arrives in Phase 2/3.
- Existing account data-deletion actions do not yet cover wellness tables (DB cascade covers
  full account deletion; the in-app "delete my data" section needs extending).
- No notification delivery exists; `wellness_milestones.notification_sent_at` and notification
  categories are forward-looking.
- `enrollment_settings`/JSONB metadata on `wellness_pathways` are minimal placeholders.

## Recommended focus for Phase 2

1. Timezone-aware "user's local day" utility (populate and use `profiles.timezone`).
2. Deterministic readiness/safety rules engine: daily + postpartum check-in red flags →
   adaptation level (`standard`/`reduced`/`recovery`/`safety_hold`) with fixed messages —
   the authoritative layer the LLM can never override.
3. `/her` route shell + enrollment flow (registry-driven onboarding sections) + middleware
   prefixes, and extend the in-app data-deletion section to wellness tables.
4. Daily check-in UI + plan rendering from `wellness_daily_plans`.
5. Apply 00007 to a dev project, regenerate types, delete the hand-written wellness rows.
